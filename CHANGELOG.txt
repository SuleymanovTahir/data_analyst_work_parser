2026-04-30 (HH OAuth через ENV)

ВАЖНО ПОМНИТЬ
- OAuth-секреты HH нельзя хранить в репозитории. Использовать только ENV: `HH_CLIENT_ID`, `HH_CLIENT_SECRET`, `HH_REDIRECT_URI`; при необходимости готовый токен можно подать через `HH_ACCESS_TOKEN`/`HH_REFRESH_TOKEN`.

ИЗМЕНЕНИЯ
- В `hh_vacancy_bot.py` добавлена поддержка HH OAuth: application token через `client_credentials`, генерация authorization URL, проверка `state`, обмен `code` на token, refresh token и безопасный public-status без раскрытия token.

2026-04-20 (HH диагностический fail-fast)

- Для HH включён предельно жёсткий безопасный режим: один HH-запрос за запуск, одна страница, без ретраев. Это не про полноту выдачи, а про быструю диагностику и явную ошибку вместо зависания.
- Если HH снова режет доступ с текущего источника, теперь это должно проявляться как короткий `403/429/timeout`, а не как минутное ожидание без ответа.

- `HH_MAX_REQUESTS_PER_RUN` снижен до `1`: backend не отправляет больше одного HH-запроса за запуск.
- Для HH отключены ретраи по умолчанию (`HH_REQUEST_RETRIES = 0`), чтобы не добивать источник повторными запросами.
- HH-таймауты ужаты ещё сильнее: отдельные HH API timeout теперь короче, а дедлайны батча и всего запуска снижены до диагностического режима.

- В web API добавлено отдельное поле `friendly_error`, чтобы сайт мог показывать короткую человеческую причину HH-сбоя без разбора raw-ошибок.

Кнопка `Дальше` у готовой второй группы и все переходы между страницами теперь пересчитывают нужную страницу по активному шаблону, а не зависят от `live_group_run` между разными serverless-инстансами. Это обход проблемы, когда `Runtime Cache` недоступен и после второй группы state теряется между `/run-now` и `/telegram-webhook`.
- HH live-анонсы упрощены: бот публикует только готовность второй группы, а страницы 3+ открываются уже из самой страницы результатов кнопками `Назад/Дальше` и номерными кнопками. Текст прогресса и итоговое сообщение обновлены под эту схему.
- Тяжёлый `hgrp:`-callback отключён из живой HH-навигации: он заново пересчитывал весь HH-поиск прямо внутри callback и подвешивал Telegram на сообщении `Загружаю страницу ...`. Кнопка `Дальше` у живой второй группы снова использует мгновенное открытие по уже собранным `vacancy_id` через `grp:`. Старые `hgrp:`-кнопки теперь не зависают, а сразу просят запустить новый поиск.
- Исправлена ошибка в callback открытия HH-группы: статусное сообщение использовало `render_run_state` до присвоения, что могло ронять обработчик сразу после открытия второй группы.
- Для HH включён безопасный режим против повторной блокировки IP: поиск теперь принудительно идёт в `1` воркер, кеш HH увеличен до `15` минут, безопасная глубина по умолчанию ограничена `1` страницей на запрос, а на один запуск действует жёсткий бюджет `HH_MAX_REQUESTS_PER_RUN`.
- Для HH добавлен circuit breaker после `403`: как только HH возвращает `403`, бот ставит HH-поиск на паузу примерно на `30` минут, сохраняет причину в state (`hh_block_until`, `hh_block_reason`) и перестаёт слать новые HH-запросы до окончания паузы.
- Ручной запуск, предпросмотр, автопоиск и web-run HH теперь сначала проверяют временный HH-блок и сразу показывают понятное сообщение `HH-поиск временно стоит на паузе...`, вместо повторных попыток добивать уже заблокированный IP.
- HH-поиск и web-preview теперь используют более понятные причины ошибок: отдельно выделяются пауза после `403` и безопасный лимит запросов за один запуск; в этих случаях бот больше не смешивает блокировку HH с текстом `Проверьте запросы...`.
- При сохранении результатов HH после поиска используется самое свежее состояние из `load_data()`, чтобы не затирать поля временного HH-блока и другие свежие изменения устаревшей копией state.
- HH-защита ужесточена ещё сильнее: повторные запросы к одному и тому же HH-эндпоинту больше не ретраятся по умолчанию, блок после `403` увеличен до `60` минут, а `429` теперь тоже рассматривается как сигнал к временной паузе HH-поиска.
- Для HH введена ротация окна запросов: один запуск проверяет только безопасную часть комбинаций (`HH_MAX_QUERY_TASKS_PER_RUN`, по умолчанию `12`), после чего курсор сдвигается в `hh_template_runtime.query_cursor`, и следующий запуск продолжает с другого участка шаблона вместо повторного перебора первых 45 запросов.
- В HH-результатах теперь явно показывается пояснение, что безопасный запуск проверил только часть комбинаций и следующий запуск продолжит дальше. Это снижает нагрузку на HH и объясняет, почему один прогон может находить меньше вакансий, чем полный тяжёлый скан.
- В web-state и runtime-status добавлено поле `hh_block_reason`, чтобы в панели и диагностике было видно не только факт паузы HH, но и её конкретную причину.
- UI HH выровнен с безопасным backend: в мастере и web-панели для HH убраны варианты `2/3/5 страниц`, потому что серверный код теперь жёстко работает только с одной страницей HH за запуск.
- HH-режим после реальных тестов смягчён: больше нет часовой паузы после первого `403`. Теперь HH ставится на паузу только после серии ограничений (`HH_BLOCK_TRIGGER_COUNT`, по умолчанию `3`), а длительность паузы уменьшена до `10` минут для `403` и `5` минут для `429`.
- HH снова работает живее: возвращены короткие retry/backoff, безопасный лимит одного запуска увеличен до `30` запросов, а окно ротации шаблона расширено до `20` комбинаций за запуск. Это даёт заметно больше вакансий за один проход без возврата к старому агрессивному полному скану.
- Старые слишком длинные HH-блоки теперь автоматически подрезаются новым guard-ом до актуального мягкого лимита, поэтому после деплоя бот не должен продолжать сидеть в устаревшей часовой паузе из старого state.
- Тексты ошибок HH обновлены под мягкий режим: одиночный `403/429` теперь описывается как частичное ограничение запроса с продолжением поиска по остальным комбинациям, а сообщение про полную HH-паузу показывается только если реально включился временный block guard.
